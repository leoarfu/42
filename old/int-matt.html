2. Spacecraft Subsystems: The Five You Must Know Cold

```

A Loft Orbital satellite (YAM-series, Longbow platform) is a small satellite with five subsystems you'll encounter in every mission lifecycle conversation. You don't need to design any of them — but you need to know what they do, what telemetry indicates health, and what anomalies look like.

ADCS

  • Attitude Determination & Control
  • Sensors: star trackers, IMUs, magnetometers, sun sensors
  • Actuators: reaction wheels, magnetorquers
  • Modes: detumbling → coarse pointing → fine pointing
  • Anomaly sign: unexpected attitude drift, wheel saturation, desaturation burns

EPS

  • Electrical Power System
  • Sources: solar panels → battery
  • Key metric: State of Charge (SoC), voltage, current per rail
  • Eclipse: battery-only operation, thermal and power budget tightens
  • Anomaly: low SoC trend, overcurrent trip on a rail → load shedding

TT&C / COMMS

  • Telemetry, Tracking & Commanding
  • S-band (uplink command, downlink housekeeping TM)
  • X-band or Ka-band for payload data downlink
  • UHF as emergency backup on many small sats
  • Anomaly: missed pass, no ack on command, anomalous RSSI

THERMAL

  • Passive: MLI blankets, coatings, radiators
  • Active: heaters (thermostatically controlled)
  • Key zones: battery (narrowest temp range), payload, propulsion
  • Eclipse is the coldest; sun-facing nadir is hottest
  • Anomaly: heater stuck ON/OFF, out-of-range component temp

OBC / AVIONICS

  • On-Board Computer: runs FSW, manages telemetry
  • Flight Software (FSW): handles modes, anomaly responses
  • Memory: SDRAM (volatile) + flash (non-volatile)
  • Watchdog timers: reset OBC if FSW hangs
  • Anomaly: unexpected reboot, memory corruption, watchdog trip

PAYLOAD

  • Customer's hardware — Loft is payload-agnostic
  • Interfaced via Hub (Loft's abstraction layer)
  • Power, data, and thermal interfaces standardised
  • CDE responsibility: customer payload health and mission objectives
  • Anomaly: unexpected power draw, no data output, thermal exceedance

Reading telemetry like a systems engineer

Telemetry is the only window into the spacecraft after launch. A CDE reading housekeeping telemetry is doing the same thing a network engineer does reading SNMP counters — pattern recognition against a known baseline, trending for degradation, alerting on threshold crossings. The vocabulary is different; the discipline is the same.

Key housekeeping parameters you'll encounter in Cockpit: battery SoC (%), bus voltage (V), individual rail currents (mA), component temperatures (°C), ADCS mode (integer state machine), wheel speeds (rpm), uplink RSSI (dBm), command counter (increments on successful command), error register (bitfield of active faults). When a customer reports "the payload isn't responding," the first thing you check is: is the satellite healthy? Power nominal? OBC alive? Comms nominal? Then narrow to the payload interface.

THE LAYER ZERO PRINCIPLE

Before diagnosing any payload or customer-facing anomaly, verify the spacecraft bus is healthy. A payload that "isn't responding" might be a payload hardware fault, a Hub interface fault, a power rail brown-out, an OBC crash, or simply a missed command due to a bad pass. Jumping to the payload-level explanation without clearing the lower layers wastes time and can lead to incorrect commanding. Matt will respect the engineer who says "first I verify the bus is nominal" before speculating about the payload.
```

3. The Loft Orbital Platform — Longbow, Hub, Cockpit

```

Understanding Loft's specific architecture matters because Matt helped build it operationally. Mixing up what the Hub does versus what the bus does, or confusing Cockpit with a generic ground system, will signal that you haven't done your homework.

The Longbow platform

Loft's current satellite bus is the Longbow — an evolution of the Airbus Arrow 150 platform, developed under an ESA Pioneer Partnership Project (IODA). It is a mid-size small satellite (~150 kg class) designed for multi-payload hosting. The bus upgrade from Arrow to Longbow increased SWaP margins, enabling larger, more power-hungry payloads such as the onboard AI computing hardware for the Altair constellation. The Longbow platform is the physical foundation that Loft holds in inventory, pre-qualified and ready for customer payloads.

The Hub

The Hub is Loft's proprietary hardware abstraction layer — a universal payload adapter that sits between the customer payload and the satellite bus. Its purpose is to standardise the interface: regardless of what the customer's payload hardware looks like, it connects to the Hub, which provides standardised interfaces for power, command, computing, and communications. This is what enables Loft to be payload-agnostic. The Hub eliminates bus-level non-recurring engineering (NRE) for each new customer — instead of custom integration every time, the Hub provides a repeatable, qualified interface. For a CDE, the Hub is the boundary layer: issues on the bus side are Loft internal; issues on the payload side are customer-facing. Understanding which side a symptom originates from is a core CDE competency.

Cockpit

Cockpit is Loft's mission control software — the ground system through which CDEs and customers interact with the spacecraft. It exposes an API (REST-based) through which commanding, telemetry retrieval, scheduling, and mission planning operations are performed. Customers interact with Cockpit to operate their payloads; CDEs use it to monitor spacecraft health, manage passes, and diagnose anomalies. Cockpit is not just a UI — it has a backend that orchestrates commanding through ground stations, manages the telemetry pipeline, and provides access control separating customer payload operations from bus health operations.

BUS vs HUB vs PAYLOAD — THE THREE-LAYER MODEL

Bus: the physical spacecraft (structure, power, ADCS, thermal, OBC, comms) — Loft's responsibility. Hub: Loft's abstraction hardware connecting bus to payload — Loft's design, customer-facing interface. Payload: customer's hardware running the mission — customer's responsibility, operated via Cockpit. When something goes wrong, the first diagnostic question is: which layer is the symptom in? A CDE who can answer this cleanly impresses Matt.

```

4. Link Budgets and RF — Your Home Territory

```

This is where you have genuine depth that Matt respects. You've done end-to-end link budget analysis at iDirect across DVB-S2 systems, Ku/Ka-band transponders, and LNBFs. The satellite side of the link is the same physics — just from the spacecraft looking down rather than the hub looking up.

The link budget equation

FREE-SPACE PATH LOSS FSPL (dB) = 20·log₁₀(d) + 20·log₁₀(f) + 20·log₁₀(4π/c) = 20·log₁₀(d_km) + 20·log₁₀(f_GHz) + 92.45 RECEIVED POWER C (dBW) = EIRP_tx (dBW) - FSPL (dB) + G/T_rx (dB/K) - k (dBW/K/Hz) - BW (dBHz) LINK MARGIN Margin (dB) = C/N₀_received - C/N₀_required Required C/N₀ = Eb/N₀_required + 10·log₁₀(data_rate_bps) ELEVATION EFFECT At low elevation (θ < 15°): FSPL increases, atmospheric attenuation increases, effective G/T drops due to higher antenna noise temperature from ground noise. Rule of thumb: 3 dB link margin reduction per 10° drop below 20° elevation.

How this maps to Loft operations

In a Loft CDE role, link budget awareness manifests in pass planning: a pass at 5° max elevation may not close the TT&C link reliably — plan commanding for passes above 15° minimum elevation. Payload downlink scheduling needs to account for elevation, weather (Ka-band rain fade), and available ground station contacts. When a customer reports low throughput or missed commands, checking whether the anomaly correlates with low-elevation passes is a first-order diagnostic step.
```

5. Pass Geometry and Ground Contact Windows

```

A LEO satellite at 550 km altitude completes an orbit in approximately 90 minutes, moving at ~7.6 km/s. From any given ground station, the satellite is above the horizon (and therefore contactable) for only a fraction of each orbit — typically 5–20 minutes per pass depending on the maximum elevation angle.

What determines a contact window

The contact window is bounded by the satellite rising above the minimum usable elevation angle at the ground station (typically 5–10° for S-band, higher for Ka-band where atmospheric losses at low elevation are significant) and setting below it at the end of the pass. Maximum elevation (the "max el" of the pass) determines the available link margin at the peak — a pass that barely scrapes 10° max elevation will have much lower peak SNR and much shorter usable window than a 70°+ overhead pass.

For a network of multiple ground stations geographically distributed, a LEO satellite at 550 km in a sun-synchronous orbit will typically have 5–15 contact opportunities per day with any given ground station, each lasting 5–12 minutes. With a global network (AWS ground stations, Loft's partner network), contact frequency is much higher — in principle, you can achieve near-continuous contact with sufficient station density.

Why this matters for CDE work

Pass scheduling is one of the foundational CDE operational activities. Commanding windows, telemetry download sessions, payload data offload, software patch uploads — all must be scheduled into available contact windows. When planning a complex operation (firmware update, payload mode change, anomaly recovery sequence), you need to know: how many passes are available in the next 24 hours, what their max elevations are, which ground stations are healthy, and whether the required command set fits within a single pass or requires multiple.

The constraint is asymmetric: a satellite with no contact window for the next 6 hours is effectively uncontrollable for that period. Any onboard autonomous response (safe mode, FDIR — Fault Detection, Isolation, and Recovery) must carry the satellite safely through the gap. This is why onboard autonomy and safe mode design matter: the spacecraft must be designed to self-protect during periods when no operator can intervene.

PASS PLANNING RULE OF THUMB

For a time-critical operation (anomaly recovery, firmware update), schedule it in the first available high-elevation pass (max el > 40°), not the next available pass regardless of geometry. A 70° pass gives you 3–4× the usable window and much higher link margin than a 15° pass. Always verify the command was received and acknowledged before the pass ends — if you lose contact mid-sequence, you need to know where the spacecraft's state machine stopped before the next contact.
```

6. Safe Mode: What It Is, When to Trigger It, What Not to Do

```

Safe mode is the spacecraft's self-preservation state — a minimal-power, maximum-safety configuration the OBC enters either autonomously (via FDIR) or by operator command when the satellite's health cannot be guaranteed in nominal operations mode. Understanding safe mode is the most important safety-relevant topic in this interview.

What safe mode does

In safe mode, the satellite sheds non-essential loads (payloads off, non-critical subsystems powered down), aligns to a power-positive attitude (solar panels facing the sun to maximise charging), activates the beacon (continuous TT&C broadcast so ground can find it), and awaits operator contact. The OBC runs minimal FSW. Battery is conserved. The satellite is in a known, recoverable state.

Safe mode is not a failure state — it is a designed, intentional recovery state. A satellite in safe mode that is healthy and communicating is under control. A satellite in nominal mode with degrading power budget and a payload drawing excess current is the more dangerous situation. Operators sometimes resist triggering safe mode because it interrupts the customer mission; the correct risk posture is that a brief planned safe mode is always preferable to an unplanned one triggered by FDIR after a worse anomaly.

When a CDE would command safe mode

You would command safe mode (or recommend it to the spacecraft systems team) when: battery SoC is trending below safe threshold with no recovery trajectory visible; a subsystem is exhibiting behaviour that is not understood and could cascade; a commanding sequence has partially executed and the satellite's state is uncertain; you are about to lose contact for an extended period and nominal mode is not stable.

The principle: when in doubt, the safest spacecraft is a boring spacecraft. Matt will respect a candidate who says "I'd always lean toward safe mode when the state is uncertain rather than try to recover in nominal mode" more than one who says "I'd keep the customer payload running as long as possible."

What never to do in an anomaly

THE THREE RULES OF SPACECRAFT COMMANDING UNDER ANOMALY

1. Never send a command you don't understand the consequence of. On orbit, commands cannot be untaken. A wrong power-off command that kills a critical subsystem may not be recoverable in the next pass. If you don't know what a command does, you don't send it. You wait for the subject matter expert.

2. Never assume the spacecraft's state — verify it from telemetry. A satellite that was in Mode A at the last pass may have transitioned autonomously since then. Before any commanding sequence, verify current state.

3. Never prioritise customer mission continuity over spacecraft health. The customer's payload data is important. The spacecraft is more important. If the bus is at risk, the payload goes off. Every time.
```

7. ConOps: What It Captures and Your Role Writing It

```

A ConOps (Concept of Operations) document describes how a mission will be operated — not what the spacecraft does physically, but how the operators, ground systems, and spacecraft work together to achieve mission objectives. Matt develops ConOps with flight operations at Loft; CDEs contribute the customer-facing operational perspective.

What a ConOps document covers

Mission phases: from pre-launch (satellite commissioning environment testing, ground system readiness) through LEOP (Launch and Early Orbit Phase — first contact, initial health check, safe mode confirmation, preliminary commissioning) through routine operations (nominal commanding cadence, telemetry monitoring, data downlink scheduling) through anomaly response (decision trees, escalation paths, contingency procedures) through end of life (deorbit planning, payload shutdown sequence, passivation).

For each phase: who does what, when, using which tools, with what authority levels, and what constitutes success or failure. A ConOps also captures constraints: minimum battery SoC before payload activation, maximum time between telemetry checks, required ground station contacts per day, thermal limits that must not be exceeded during extended payload operations.

Your CDE contribution to ConOps

As a CDE, you contribute the customer operational requirements: when the customer needs the payload operational, at what duty cycle, with what data latency, and with what notification in case of anomaly. You translate customer requirements into operational constraints for the spacecraft team. If the customer needs payload operations for 14 consecutive orbits each day, you need to verify that the power budget and thermal budget support it, and work with Matt's team to establish the operational envelope. The ConOps is the contract between what the customer needs and what the spacecraft can provide.
```

```
Exercise 1 — Satellite Link Budget Calculator

Compute a complete downlink and uplink budget for a small satellite, including free-space path loss, rain fade, elevation-dependent atmospheric losses, and link margin. Show how margin changes across a pass as elevation varies.

``` 
import numpy as np
import matplotlib.pyplot as plt

# ── Constants ──────────────────────────────────────────────────────────────
c = 3e8          # speed of light m/s
k_B = 1.38e-23   # Boltzmann constant J/K
k_dB = 10 * np.log10(k_B)  # ≈ -228.6 dBW/K/Hz

# ── Free-space path loss (dB) ───────────────────────────────────────────────
def fspl_db(freq_ghz: float, dist_km: float) -> float:
    """FSPL = 20log(d) + 20log(f) + 92.45  (d in km, f in GHz)"""
    return 20*np.log10(dist_km) + 20*np.log10(freq_ghz) + 92.45

# ── Slant range from elevation angle ───────────────────────────────────────
Re = 6371.0  # km

def slant_range_km(alt_km: float, elev_deg: float) -> float:
    """
    Slant range from ground station to satellite at altitude alt_km,
    given ground station elevation angle elev_deg.
    """
    eps = np.radians(elev_deg)
    rho = np.arcsin((Re / (Re + alt_km)) * np.cos(eps))
    lam = np.pi/2 - eps - rho
    return Re * np.sin(lam) / np.cos(eps)  # law of sines on Earth triangle

# ── Atmospheric attenuation (simplified ITU-R P.676) ──────────────────────
def atmo_loss_db(freq_ghz: float, elev_deg: float) -> float:
    """
    Approximate atmospheric loss vs elevation and frequency.
    Based on ITU-R P.676 simplified model for clear-sky gas absorption.
    """
    # Zenith attenuation (rough model)
    if freq_ghz < 3:
        zenith_loss = 0.05   # S-band, very low
    elif freq_ghz < 10:
        zenith_loss = 0.10   # X-band
    elif freq_ghz < 20:
        zenith_loss = 0.20   # Ku-band
    else:
        zenith_loss = 0.50   # Ka-band

    # Scale by sin(elevation) — low elevation = more atmosphere traversed
    elev_rad = np.radians(max(elev_deg, 2.0))
    return zenith_loss / np.sin(elev_rad)

# ── Rain fade (simplified CDF-based model) ────────────────────────────────
def rain_fade_margin_db(freq_ghz: float, rain_rate_mm_hr: float = 5.0) -> float:
    """
    Approximate rain fade for 99.5% availability design.
    Scales with frequency and rain rate.
    """
    if freq_ghz < 4:
        return 0.0   # L/S-band: negligible rain fade
    # Simplified power-law approximation
    alpha = 0.0001 * (freq_ghz ** 1.8) * (rain_rate_mm_hr ** 0.6)
    return alpha * 5.0  # dB over 5 km effective rain path

# ── Complete link budget ────────────────────────────────────────────────────
def link_budget(
    freq_ghz: float,
    sat_alt_km: float,
    elev_deg: float,
    tx_eirp_dbw: float,
    rx_gt_db_k: float,
    data_rate_bps: float,
    req_eb_n0_db: float,
    rain_rate_mm_hr: float = 5.0,
) -> dict:
    """
    Full downlink budget.
    Returns a dict with all intermediate values and final margin.
    """
    d_km = slant_range_km(sat_alt_km, elev_deg)
    fspl = fspl_db(freq_ghz, d_km)
    atmo = atmo_loss_db(freq_ghz, elev_deg)
    rain = rain_fade_margin_db(freq_ghz, rain_rate_mm_hr)

    # Received C/N0 (dBHz)
    c_n0 = tx_eirp_dbw - fspl - atmo - rain - k_dB + rx_gt_db_k

    # Required C/N0 for the data rate
    req_c_n0 = req_eb_n0_db + 10*np.log10(data_rate_bps)

    margin = c_n0 - req_c_n0

    return {
        "freq_ghz":        freq_ghz,
        "elevation_deg":   elev_deg,
        "slant_range_km":  d_km,
        "fspl_db":         fspl,
        "atmo_loss_db":    atmo,
        "rain_fade_db":    rain,
        "tx_eirp_dbw":     tx_eirp_dbw,
        "rx_gt_db_k":      rx_gt_db_k,
        "c_n0_dbhz":       c_n0,
        "req_c_n0_dbhz":   req_c_n0,
        "link_margin_db":  margin,
    }

# ── Scenario: Loft-style S-band TT&C downlink ──────────────────────────────
# Ground station uplink TO satellite (commanding)
# Satellite: S-band receiver, omnidirectional antenna ~ -5 dB/K G/T
# Ground station: 5m dish, 2kW EIRP ≈ +62 dBW at S-band
# Data rate: 9600 bps command uplink, BPSK, req Eb/N0 = 10.5 dB

scenario_ttc = {
    "freq_ghz":       2.2,
    "sat_alt_km":     550,
    "tx_eirp_dbw":    62.0,     # large ground station
    "rx_gt_db_k":    -5.0,      # satellite omni receive antenna
    "data_rate_bps":  9600,
    "req_eb_n0_db":   10.5,
    "rain_rate_mm_hr": 3.0,
}

elevations = np.linspace(5, 85, 80)

```
Exercise 2 — Spacecraft Telemetry Health Monitor

Build a subsystem health triage tool that takes a dictionary of telemetry values and evaluates each subsystem against its nominal ranges, producing a structured health report with alert levels. This is the logic you'd write to automate Cockpit telemetry monitoring.

``` 
from dataclasses import dataclass, field
from enum import Enum
from typing import Any
import json
from datetime import datetime

class AlertLevel(Enum):
    NOMINAL  = "NOMINAL"
    CAUTION  = "CAUTION"   # out of yellow limits — monitor
    WARNING  = "WARNING"   # out of red limits — act soon
    CRITICAL = "CRITICAL"  # safety threshold — act now

@dataclass
class TelemetryLimit:
    name: str
    unit: str
    green_lo: float
    green_hi: float
    yellow_lo: float   # caution below this
    yellow_hi: float   # caution above this
    red_lo: float      # warning below this
    red_hi: float      # warning above this

    def evaluate(self, value: float) -> tuple[AlertLevel, str]:
        if value < self.red_lo or value > self.red_hi:
            return AlertLevel.CRITICAL, f"{value:.2f} {self.unit} — OUTSIDE RED LIMITS [{self.red_lo}, {self.red_hi}]"
        elif value < self.yellow_lo or value > self.yellow_hi:
            return AlertLevel.WARNING, f"{value:.2f} {self.unit} — outside yellow limits [{self.yellow_lo}, {self.yellow_hi}]"
        elif value < self.green_lo or value > self.green_hi:
            return AlertLevel.CAUTION, f"{value:.2f} {self.unit} — outside green limits [{self.green_lo}, {self.green_hi}]"
        else:
            return AlertLevel.NOMINAL, f"{value:.2f} {self.unit} ✓"

# ── Define subsystem limit tables ──────────────────────────────────────────
LIMITS = {
    # EPS
    "battery_soc_pct":     TelemetryLimit("Battery SoC", "%",  60, 100, 40, 100, 20, 100),
    "bus_voltage_v":       TelemetryLimit("Bus Voltage",  "V",  7.0, 8.4, 6.8, 8.5, 6.5, 8.7),
    "solar_current_ma":    TelemetryLimit("Solar Current","mA", 200, 1500, 100, 1600, 50, 1800),
    "payload_current_ma":  TelemetryLimit("Payload Rail", "mA", 0, 800, 0, 900, 0, 1000),

    # THERMAL
    "battery_temp_c":      TelemetryLimit("Battery Temp", "°C", -10, 30, -15, 35, -20, 40),
    "payload_temp_c":      TelemetryLimit("Payload Temp", "°C", -20, 50, -25, 55, -30, 65),
    "obc_temp_c":          TelemetryLimit("OBC Temp",     "°C", -20, 60, -25, 65, -30, 70),

    # ADCS
    "wheel_speed_rpm":     TelemetryLimit("Wheel Speed",  "rpm", -3000, 3000, -4500, 4500, -5000, 5000),
    "adcs_mode":           TelemetryLimit("ADCS Mode",    "int",  2, 5, 1, 6, 0, 7),  # 0=off, 1=detumble, 2=coarse, 3=fine, etc.

    # COMMS
    "uplink_rssi_dbm":     TelemetryLimit("Uplink RSSI",  "dBm", -80, -20, -90, -20, -100, -20),
    "cmd_counter":         TelemetryLimit("Cmd Counter",  "count", 0, 1e6, 0, 1e6, 0, 1e6),  # just tracks

    # OBC
    "obc_uptime_hrs":      TelemetryLimit("OBC Uptime",   "hrs",  0, 1e6, 0, 1e6, 0, 1e6),  # info only
    "mem_usage_pct":       TelemetryLimit("Memory Usage", "%",    0, 70, 70, 85, 85, 95),
}

@dataclass
class SubsystemReport:
    name: str
    parameters: dict[str, tuple[AlertLevel, str]] = field(default_factory=dict)

    @property
    def worst_level(self) -> AlertLevel:
        levels = [lvl for lvl, _ in self.parameters.values()]
        priority = [AlertLevel.CRITICAL, AlertLevel.WARNING, AlertLevel.CAUTION, AlertLevel.NOMINAL]
        for p in priority:
            if p in levels:
                return p
        return AlertLevel.NOMINAL

def evaluate_telemetry(telemetry: dict[str, float]) -> dict[str, SubsystemReport]:
    """
    Evaluate a telemetry snapshot against all defined limits.
    Returns a dict of subsystem reports.
    """
    subsystems = {
        "EPS":    ["battery_soc_pct", "bus_voltage_v", "solar_current_ma", "payload_current_ma"],
        "THERMAL":["battery_temp_c", "payload_temp_c", "obc_temp_c"],
        "ADCS":   ["wheel_speed_rpm", "adcs_mode"],
        "COMMS":  ["uplink_rssi_dbm", "cmd_counter"],
        "OBC":    ["obc_uptime_hrs", "mem_usage_pct"],
    }

    reports = {}
    for sys_name, params in subsystems.items():
        report = SubsystemReport(name=sys_name)
        for p in params:
            if p in telemetry and p in LIMITS:
                level, msg = LIMITS[p].evaluate(telemetry[p])
                report.parameters[p] = (level, msg)
        reports[sys_name] = report
    return reports

def print_health_report(telemetry: dict, timestamp: str = None):
    timestamp = timestamp or datetime.utcnow().isoformat()
    reports = evaluate_telemetry(telemetry)

    icons = {
        AlertLevel.NOMINAL:  "✓",
        AlertLevel.CAUTION:  "⚠",
        AlertLevel.WARNING:  "⚡",
        AlertLevel.CRITICAL: "🔴",
    }

    print(f"\n{'='*62}")
    print(f"  SPACECRAFT HEALTH REPORT  ·  {timestamp}")
    print(f"{'='*62}")

    for sys_name, report in reports.items():
        icon = icons[report.worst_level]
        print(f"\n  {icon}  {sys_name} — {report.worst_level.value}")
        for param, (level, msg) in report.parameters.items():
            p_icon = icons[level]
            limit = LIMITS[param]
            print(f"      {p_icon}  {limit.name:<20} {msg}")

    # Overall status
    all_levels = [r.worst_level for r in reports.values()]
    priority = [AlertLevel.CRITICAL, AlertLevel.WARNING, AlertLevel.CAUTION, AlertLevel.NOMINAL]
    overall = next((p for p in priority if p in all_levels), AlertLevel.NOMINAL)
    print(f"\n{'='*62}")
    print(f"  OVERALL: {overall.value}  {icons[overall]}")
    print(f"{'='*62}\n")

    return overall
