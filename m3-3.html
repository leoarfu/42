Payload Architectures:
Transparent vs Regenerative

MODULE 3
COMMS PAYLOAD

The communications payload is the reason the satellite exists. The payload architecture — how it handles the signal between receive and transmit — is the single decision with the largest downstream consequences for capacity, latency, flexibility, and cost. This lesson maps the spectrum from transparent bent-pipe through digital channeliser to fully regenerative processing, explains what each architecture can and cannot do, and identifies where LEO broadband constellations sit on it and why.

1. The Transparent (Bent-Pipe) Repeater

The transparent payload amplifies and frequency-converts the signal without demodulating it — the satellite sees RF power, not bits. Whatever modulation, coding, and data format the ground transmits passes through unchanged.

No demodulation at any stage. The satellite amplifies and frequency-shifts; the signal content is never inspected.

Strengths

Simplicity and heritage: transparent payloads have been the standard for GEO commercial satellites for 60 years — well-understood, reliable, straightforwardly testable. Waveform agnosticism: because the satellite never demodulates, it is compatible with any modulation and coding scheme the ground uses. Upgrading the air interface requires only ground terminal software changes, not satellite hardware changes.

Limitations

No on-board switching: traffic received on one beam must go to a predetermined downlink channel — the payload cannot route between beams. Noise accumulation: uplink noise is amplified and transmitted on the downlink; the noise budgets are coupled. Gateway colocation dependency: every user link must be paired with a gateway link within the same satellite footprint at the same time. No gateway visible to the satellite means no service.

THE GATEWAY COLOCATION PROBLEM For a transparent LEO constellation, user traffic comes down to a nearby gateway, routes through the ground network, and goes back up to the satellite for retransmission to the user. If no gateway is within the satellite's coverage cap when the user needs service — over an ocean, for example — the link cannot be established. This is the architectural constraint that regenerative payloads with ISLs eliminate.

2. The Regenerative (Processing) Payload

A regenerative payload demodulates the received signal, processes the bits, and remodulates a new signal for transmission. The satellite operates as a full network node rather than a transparent relay.

Highlighted stages convert signal to bits and back — the satellite sees and processes digital data.

Strengths

On-board switching and routing: the DBP can switch traffic between any uplink and downlink beam, and forward traffic via ISL to the next satellite. Noise decoupling: demodulation regenerates a clean signal; the downlink starts with a fresh noise budget independent of uplink noise. Flexible capacity allocation: the DBP can dynamically reallocate bandwidth, power, and beam pointing across beams in response to real-time demand.

Limitations

Complexity and cost: a high-throughput DBP is a demanding, power-hungry piece of radiation-hardened hardware. Waveform lock-in: changing the air interface post-launch requires changing on-board software at minimum, which is more constrained than a ground-only change. Development and test burden: verifying a DBP under all traffic patterns and interference conditions is significantly harder than verifying a transparent amplifier chain.

3. The Digital Channeliser

Between the two poles lies the digital channeliser: the received signal is digitised by an ADC, processed by an FFT filter bank into narrowband sub-channels that can be routed and recombined independently, then converted back to analogue by a DAC and amplified. No full demodulation to bits — the satellite sees frequency sub-bands, not packets.

Advantages over a traditional transparent payload: flexible bandwidth allocation (sub-channels can be combined in software to create beams of any width); sub-band routing (frequency sub-bands from one uplink beam can go to a different downlink beam); interference notching (known interfering signals can be digitally suppressed before re-amplification). This is the architecture used by SES O3b mPower and several VHTS GEO platforms.

4. Where LEO Broadband Constellations Sit

Starlink uses a regenerative payload with on-board routing. This enables the ISL mesh routing that is central to Starlink's architecture — without regeneration, ISL routing is essentially impossible because you cannot switch bitstreams at the satellite if you never convert to bits. The DBP is a proprietary SpaceX design.

OneWeb (first generation) uses a transparent payload. All routing is performed on the ground through the gateway network. Simpler and cheaper in the space segment but creates a tight gateway colocation dependency over oceanic and remote coverage areas.

Telesat Lightspeed is designed as fully regenerative from the outset, with on-board switching, dynamic beam management, and optical ISLs. The MDA Aurora platform is software-defined — payload functionality is programmable on orbit — representing the leading edge of commercial regenerative payload design.

Kuiper has not publicly confirmed its on-board processing architecture in detail, but its FCC filings indicate advanced digital processing capability and ISL support.

THE ARCHITECTURE CHOICE FOLLOWS THE NETWORK DESIGN Starlink's ISL mesh requires regeneration — you can't route across an ISL without bits. OneWeb's gateway-centric ground network makes a transparent payload viable, at the cost of geographic gateway density. The payload architecture is not chosen independently; it is determined by the network topology decision made at the system level.

5. The Software-Defined Payload

The most recent development is the software-defined satellite (SDS) — a regenerative architecture where payload functionality is defined in software running on radiation-hardened programmable hardware (FPGAs or general-purpose processors), rather than fixed ASICs. Post-launch software uploads can change the waveform, channel plan, beam configuration, and routing logic without any physical change to the satellite.

This provides post-launch reconfigurability (respond to market changes, interference environments, or technology upgrades), multi-mission capability (same hardware serves different customers by changing software), and rapid iteration (software updates replace hardware re-spins). Telesat's MDA Aurora and Airbus's OneSat GEO platform both take this approach.

The fundamental limitation: the programmable hardware must be specified at launch with enough capacity to support future software functionality. You cannot add processing capacity that was not physically present. Designing in sufficient headroom without massively oversizing the hardware is the key architectural challenge.

6. On-Board Processing Power Requirements

The digital baseband processor in a regenerative satellite is the payload's most power-hungry component after the high-power amplifiers. A DBP handling 100 Gbps of aggregate throughput requires on the order of 500 W–2 kW depending on architecture efficiency — a significant fraction of the satellite's total power budget and a direct driver of solar array area and battery sizing.

The trend is toward more capable, lower-power ASICs, driven by the same semiconductor technology improvements that drove terrestrial data centre efficiency gains. Each generation of radiation-hardened ASIC technology (typically 1–2 generations behind commercial process nodes due to qualification requirements) delivers more compute per watt. This trend is what made fully regenerative architectures practical for mass-produced LEO constellation satellites in the early 2020s — a decade earlier, the power and mass budget for a high-capacity DBP was prohibitive at the 150–500 kg satellite mass point that LEO broadband economics require.

COMPUTE PER WATT — THE ENABLING TREND The same Moore's Law curve that drove smartphone processing into wearables is driving satellite on-board processing into mass-produced LEO spacecraft. Radiation-hardened ASICs lag the commercial frontier by 5–10 years, but the direction is the same: more function per watt per dollar per gram. This is why regenerative payloads that were impractical at 100 kg satellites in 2015 are standard on 300 kg satellites in 2025.

Exercise 1 — Transparent vs Regenerative Noise Budget

Compare end-to-end C/N for a transparent relay versus a regenerative relay where the downlink noise budget is independent of the uplink.

import math

def db_to_lin(db): return 10 ** (db / 10)
def lin_to_db(x):  return 10 * math.log10(x)

def transparent_cn(ul_db, dl_db):
    """End-to-end C/N for a transparent relay: 1/(1/CN_ul + 1/CN_dl)."""
    return lin_to_db(1 / (1/db_to_lin(ul_db) + 1/db_to_lin(dl_db)))

print("Transparent relay — end-to-end C/N (downlink fixed at 15 dB)\n")
print(f"{'Uplink C/N':>12}{'End-to-end C/N':>16}{'Penalty vs DL':>16}")
print("-" * 46)
for ul in [5, 8, 10, 12, 15, 20, 25]:
    e2e = transparent_cn(ul, 15.0)
    print(f"{ul:>10}dB{e2e:>14.1f}dB{15-e2e:>14.1f}dB")

print("\nRegenerative relay: downlink C/N = 15 dB regardless of uplink")
print("(provided uplink closes above demod threshold ~5-7 dB with FEC)")

Exercise 2 — Digital Channeliser Sub-Band Allocation

Model demand-proportional sub-band allocation and compare efficiency against a fixed transparent channel plan.

def channeliser(total_mhz, n_sc, demands):
    sc_bw = total_mhz / n_sc
    total = sum(demands)
    allocs = []
    for i, d in enumerate(demands):
        n = round(d / total * n_sc)
        allocs.append((i+1, d, n, n*sc_bw))
    return allocs, sc_bw

demands = [10, 30, 15, 5, 25, 15]   # relative demand, 6 beams
allocs, sc_bw = channeliser(500, 20, demands)

print(f"Digital Channeliser: 500 MHz / 20 sub-channels of {sc_bw:.0f} MHz\n")
print(f"{'Beam':>5}{'Demand':>9}{'Sub-chans':>11}{'Alloc (MHz)':>13}{'%':>7}")
print("-" * 47)
for beam, dem, nsc, bw in allocs:
    print(f"{beam:>5}{dem:>9}{nsc:>11}{bw:>12.0f} {bw/500*100:>6.1f}%")

# Fixed transparent payload: 6 equal channels of 83.3 MHz each
fixed_bw = 500 / 6
print(f"\nFixed transparent: each beam gets {fixed_bw:.1f} MHz regardless of demand")
total_demand = sum(demands)
for i, d in enumerate(demands):
    want = d / total_demand * 500
    waste = fixed_bw - want if fixed_bw > want else 0
    unmet = want - fixed_bw if want > fixed_bw else 0
    print(f"Beam {i+1}: demand wants {want:.0f} MHz, gets {fixed_bw:.0f} MHz "
          f"→ {'waste' if waste>0 else 'UNMET'} {max(waste,unmet):.0f} MHz")

Exercise 3 — Gateway Colocation Analysis

For a transparent LEO constellation, check which gateways are simultaneously visible to the same satellite serving a given user location.

import math

RE = 6378.137

def cap_angle(alt_km, min_elev_deg):
    r = RE + alt_km
    e = math.radians(min_elev_deg)
    return math.degrees(math.acos(max(-1., min(1., RE*math.cos(e)/r))) - e)

def arc_deg(lat1, lon1, lat2, lon2):
    p1=math.radians(lat1); l1=math.radians(lon1)
    p2=math.radians(lat2); l2=math.radians(lon2)
    c=math.sin(p1)*math.sin(p2)+math.cos(p1)*math.cos(p2)*math.cos(l2-l1)
    return math.degrees(math.acos(max(-1., min(1., c))))

gateways = [
    ("UK-Goonhilly",     50.0,  -5.2),
    ("US-Virginia",      38.9, -77.5),
    ("Portugal-Lisbon",  38.7,  -9.1),
    ("Brazil-Sao Paulo",-23.5, -46.6),
    ("Canada-Montreal",  45.5, -73.6),
]

for user_lat, user_lon, label in [
    (45.0, -35.0, "Mid-Atlantic"),
    (60.0, -20.0, "North Atlantic"),
    (51.5,  -0.1, "London"),
]:
    rho = cap_angle(550, 10)
    visible = [(n, arc_deg(user_lat, user_lon, gl, glo))
               for n, gl, glo in gateways
               if arc_deg(user_lat, user_lon, gl, glo) <= rho]
    print(f"User: {label} ({user_lat}°N {abs(user_lon)}°W)  "
          f"Cap radius: {rho:.1f}°")
    if visible:
        for n, d in visible:
            print(f"  ✓ {n}  ({d:.1f}° away)")
    else:
        print("  ✗ No gateway visible — transparent payload cannot serve this user")
        print("    Regenerative + ISL would route to satellite with gateway in view")
    print()
