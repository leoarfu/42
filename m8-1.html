The Gateway & the Feeder Link

MODULE 8
GROUND SEGMENT & OPERATIONS
FOUNDATION

Module 7 ended at the gateways where the mesh touches down. This module builds them — and everything else it takes to keep a constellation running. It opens a new phase of the program: the first six modules built the constellation as a machine that moves bits through space; this phase asks what it takes to operate that machine — the ground network that connects it to the terrestrial internet, and the operations that keep a fleet of thousands of satellites healthy. The first question is the most basic: how does a packet, having crossed the mesh, actually reach the internet? The answer is the gateway — a large ground station that connects the space network to terrestrial fibre — and the feeder link, the high-capacity radio trunk between a satellite and a gateway that is a fundamentally different animal from the user link this program has studied until now.

The feeder link is the user link's opposite in almost every respect, and this lesson quantifies why. Where a user terminal is small and cheap with a G/T of 11 to 21 dB/K (Module 6), a gateway is a large engineered dish — a 7.3 m antenna delivers a G/T of ~41 dB/K, some 26 dB better — because a gateway is a fixed, powered, professionally-sited installation with none of the user terminal's constraints. Where a user beam carries one user's traffic over a few hundred megahertz, the feeder link carries the aggregate of many user beams — 32 beams of 250 MHz each is 8 GHz of trunk bandwidth — because every user a satellite serves must have its traffic funnelled down to the ground somewhere. The feeder link is the mesh's on-ramp to the internet, and it is engineered as a high-capacity backhaul trunk, not a consumer access link. The spine of this module, a GroundSegment model, begins here with the feeder-link budget and the gateway's capacity.

1. What a gateway is

A gateway (or ground station, or teleport) is the physical point where the satellite network connects to the terrestrial internet. It is a large antenna — typically several metres of parabolic dish, often several such dishes per site — with high-power transmitters, cooled low-noise receivers, and a fibre connection to a terrestrial point of presence (m8-3). Its job is to relay traffic between the satellites overhead and the ground network: user traffic coming down from the constellation is handed to terrestrial fibre and routed to its internet destination; traffic coming up from the internet is sent to the constellation for delivery to users.

A gateway is everything a user terminal is not. It is fixed, so it can use a large dish that would be absurd on a rooftop or a fuselage (Module 6's whole trade). It is powered from the grid, so it can run high-power amplifiers and cooled receivers. It is professionally sited and maintained, so it can be placed where the sky is clear and the backhaul is good. And it tracks the fast-moving LEO satellites with a steerable mount — often multiple dishes so that one can acquire the next satellite while another still serves the current one (the m6-5 make-before-break pattern, at gateway scale). The gateway is the constellation's expensive, capable, terrestrial anchor, and there are far fewer of them than there are users — which is the whole point of a mesh (m7-1): concentrate the ground infrastructure at a handful of high-capacity sites.

2. The feeder link versus the user link

The radio link between a satellite and a gateway is the feeder link, and it differs from the user link (Module 4, the satellite-to-terminal link) in three defining ways:

It aggregates. A user link carries one user's traffic. A feeder link carries the combined traffic of every user beam the satellite is serving — potentially dozens of beams, each hundreds of megahertz wide. The feeder is a trunk, not an access link: it funnels the satellite's entire served capacity down to one gateway (or splits it across a few), so it must be far higher in capacity than any single user link. This is why feeder links use the widest available spectrum allocations — Ka-band (20/30 GHz), and increasingly Q/V-band (40/50 GHz) — where gigahertz of bandwidth can be found.

It has a capable ground end. The user link is constrained by the user terminal's small aperture (Module 6). The feeder link's ground end is a large gateway dish with a high G/T and high EIRP, so the ground side of the feeder link is enormously capable — the link's limiting end is the satellite, not the ground. This inverts the user-link intuition, where the terminal was the weak end; on the feeder link, the gateway is the strong end and carries the link.

It is engineered, not deployed. A user terminal is installed by a consumer or an installer in whatever location the user happens to be (Module 6's segments). A gateway is sited — placed deliberately where the rain climate, the backhaul, and the regulatory access are favourable (m8-3). This freedom to choose the location is what makes the feeder link's central problem — rain outage — solvable through site diversity (m8-2), an option the user link never had.

The feeder link, then, is a high-capacity, high-G/T, deliberately-sited backhaul trunk. It uses the same link-budget physics as Module 4, but every parameter is pushed to the opposite extreme from the user link.

3. The feeder-link budget

The Module 4 link equation, applied at gateway scale. Consider the feeder downlink (satellite to gateway) at Ka-band. The gateway's antenna gain, for a 7.3 m dish at 20 GHz with 65% efficiency:

G = η·(πD/λ)² = 61.8 dBi

and with a cooled receiver (system temperature ~120 K), the gateway's G/T is:

G/T = 61.8 − 10·log₁₀(120) = ~41 dB/K

against a user terminal's 11–21 dB/K (Module 6) — the gateway is 20 to 30 dB more sensitive. The slant range at a 20° elevation (gateways track to low elevation from clear sites) is ~1,294 km, giving a free-space loss of ~180.7 dB at 20 GHz (~186.7 dB at 40 GHz — the price of Q/V-band's extra bandwidth is ~6 dB more path loss). The enormous gateway G/T means the feeder downlink runs at very high signal-to-noise ratio in clear sky — the link closes with margin to spare and rides the top of the MODCOD ladder, exactly as the m5-7 saturation analysis predicts for a high-G/T terminal. The feeder link's clear-sky capacity is therefore limited not by the link budget but by the bandwidth — which is why the game is to find as much spectrum as possible.

4. The feeder link's capacity

Because the gateway's G/T saturates the MODCOD ladder in clear sky, the feeder link's capacity is set by its bandwidth and its high-SNR spectral efficiency. Aggregating 32 user beams of 250 MHz each demands 8 GHz of feeder bandwidth, and at a feeder SNR of 15–20 dB (comfortable for a big-dish gateway) the spectral efficiency is 5–6.7 bit/s/Hz, giving:

feeder capacity ≈ 5–6.7 bit/s/Hz × 8 GHz ≈ 40–53 Gbps per feeder link

per satellite-to-gateway connection. This is the number that matters for the ground segment's sizing: a satellite serving tens of gigabits of user traffic needs a feeder link of comparable capacity to carry it down, and a gateway serving several satellites at once needs a total capacity — and a fibre backhaul (m8-3) — of hundreds of gigabits. The feeder link is where the constellation's aggregate traffic is concentrated, and its capacity is the ground segment's fundamental throughput unit. Where the user link's story was G/T and availability (Module 6), the feeder link's story is bandwidth and aggregation: how much spectrum, how many beams funnelled through it, how many gigabits per gateway.

5. Why the feeder link is not just a big user link

It would be tempting to treat the feeder link as merely a user link with a bigger antenna, but three differences make it a distinct engineering problem. First, the aggregation changes the failure consequence: a user link failing drops one user; a feeder link failing drops every user whose traffic that satellite was funnelling through it — the feeder is a concentration point, and its reliability requirement is correspondingly higher (m8-2's site diversity exists because of this). Second, the frequency band changes the physics: pushing to Q/V-band for bandwidth means far worse rain attenuation (Module 4's rain model gets dramatically worse at 40/50 GHz), so the feeder link's availability problem is more severe than any user link's — the very bandwidth that makes the feeder capable makes it fragile to weather. Third, the siting freedom changes the solution space: because a gateway can be placed anywhere, the feeder link's rain problem is solved geographically (put gateways where it rains less, and use several so rain never hits all at once) rather than through link margin alone — a solution the user link, stuck wherever the user is, could never use.

These three — concentration, band-driven fragility, and siting freedom — are why the ground segment is its own module rather than a footnote to Module 4. The feeder link is the user link's physics turned to the opposite extreme and wrapped in an entirely different set of constraints, and the rest of Part I follows the consequences: the availability problem the concentration and the band create (m8-2), and the placement problem the siting freedom opens (m8-3).

6. The gateway in the mesh

The feeder link connects to the mesh Module 7 built. A packet from a user climbs to the serving satellite (m6-5 access), crosses the mesh to a satellite over a gateway (m7-4 routing), descends the feeder link to that gateway, and enters terrestrial fibre. The gateway is the mesh's egress point — the place m7-7 identified as "where the mesh touches down" — and how many gateways there are, and where, determines how the mesh's traffic reaches the ground. A mesh with abundant gateways can egress traffic near its destination; a mesh with few gateways must carry traffic further across the ISLs before it can descend. The gateway count and placement is therefore a joint optimisation with the mesh routing (m8-3), and the feeder link is the interface between the two: the mesh delivers traffic to a satellite over a gateway, and the feeder link carries it the last hop to the ground.

This is the m7-7 export made concrete. Module 7 handed forward "the gateways where the mesh touches down" as a constraint; this lesson defines what a gateway is and what the feeder link that connects it must do; and the rest of the module sizes, places, and operates the ground network that results. The feeder link is the seam between the space segment and the ground segment, and getting it right — enough capacity, enough availability, in the right places — is what the ground segment is for.

8. What this lesson delivers to the design chain

The spine begins as a FeederLink and Gateway model: the feeder-link budget (gateway G/T, EIRP, FSPL at the chosen band), the feeder capacity from bandwidth and high-SNR spectral efficiency, and the gateway as an aggregation point with a capacity and a fibre backhaul requirement. It takes the m7-7 delivered-traffic profile — how much traffic the mesh needs to egress, and roughly where — and turns it into a feeder-capacity requirement: how many gigabits each gateway must carry, and therefore how capable each feeder link must be. The gateway's clear-sky capacity is now a computed number.

What the clear-sky budget cannot yet tell us is whether the gateway is available. A feeder link at Ka or Q/V-band is savaged by rain, and a single gateway cannot achieve the availability a backhaul trunk demands — the concentration point that carries a whole satellite's traffic cannot be allowed to fail every time it rains. The answer is site diversity, and it is m8-2: the feeder link's availability problem, and the geographic solution that the gateway's siting freedom makes possible. Only with availability solved can the gateways be placed (m8-3) and the ground network sized.

Exercise 1 — The feeder-link budget

Build the gateway's link and contrast it with the user link.

import math
db = lambda x: 10*math.log10(x) if x > 0 else -999.0
Re, h = 6371.0, 550.0

def dish_gain_db(D_m, f_ghz, eta=0.65):
    lam = 0.3/f_ghz
    return db(eta*(math.pi*D_m/lam)**2)

def slant_km(elev_deg, alt=h):
    e = math.radians(elev_deg)
    return Re*(math.sqrt(((Re+alt)/Re)**2 - math.cos(e)**2) - math.sin(e))

def fspl_db(range_km, f_ghz):
    return 92.45 + 20*math.log10(range_km) + 20*math.log10(f_ghz)

# gateway G/T at several dish sizes and bands
print(f"{'dish':>5}{'band':>7}{'gain dBi':>10}{'G/T dB/K':>10}")
for D in [3.0, 5.0, 7.3, 9.0]:
    for f, Tsys in [(20, 120), (40, 150)]:

Exercise 2 — Feeder capacity and aggregation

Size the trunk against the user beams it carries.

import math

def feeder_capacity_gbps(bandwidth_ghz, snr_db):
    eff = math.log2(1 + 10**(snr_db/10))
    return eff * bandwidth_ghz  # Gbps (GHz × bit/s/Hz)

def aggregate_bandwidth_ghz(n_beams, beam_mhz):
    return n_beams * beam_mhz / 1000

for n_beams in [8, 16, 32, 48]:
    bw = aggregate_bandwidth_ghz(n_beams, 250)
    cap = feeder_capacity_gbps(bw, 18)
    print(f"{n_beams:>2} beams -> {bw:.1f} GHz feeder, "
          f"{cap:.0f} Gbps @ 18 dB SNR")

def slant_km(elev_deg, alt=h):
    e = math.radians(elev_deg)
    return Re*(math.sqrt(((Re+alt)/Re)**2 - math.cos(e)**2) - math.sin(e))

class Gateway:
    def __init__(self, dish_m=7.3, band_ghz=20, tsys_k=120, eta=0.65):
        self.dish_m, self.band, self.tsys = dish_m, band_ghz, tsys_k
        lam = 0.3/band_ghz
        self.gain_db = db(eta*(math.pi*dish_m/lam)**2)
        self.gt_db = self.gain_db - db(tsys_k)

class FeederLink:
    def __init__(self, gateway, bandwidth_ghz=8.0, elev_deg=20):
        self.gw = gateway
        self.bw = bandwidth_ghz
        self.range_km = slant_km(elev_deg)
        self.fspl_db = 92.45 + 20*math.log10(self.range_km) \
                       + 20*math.log10(gateway.band)

    def capacity_gbps(self, snr_db=18):
        eff = math.log2(1 + 10**(snr_db/10))
        return eff * self.bw

    def aggregates_beams(self, beam_mhz=250):
        return int(self.bw*1000 / beam_mhz)

    def report(self):
        print(f"  gateway: {self.gw.dish_m} m, {self.gw.band} GHz, "
              f"G/T {self.gw.gt_db:.1f} dB/K")
        print(f"  feeder:  {self.bw} GHz, range {self.range_km:.0f} km, "
              f"FSPL {self.fspl_db:.1f} dB")
        print(f"  capacity: {self.capacity_gbps():.0f} Gbps "
              f"(~{self.aggregates_beams()} user beams)")

gw = Gateway()
fl = FeederLink(gw)
print("── FEEDER LINK (Ka-band, 7.3 m gateway) ──")
fl.report()

# size against the mesh's delivered traffic (m7-7 export)
mesh_egress_gbps = 120   # a satellite's served traffic needing to reach ground
n_feeders = math.ceil(mesh_egress_gbps / fl.capacity_gbps())
print(f"\n  mesh egress {mesh_egress_gbps} Gbps -> "
      f"{n_feeders} feeder link(s) needed")

REFLECTION

Four prompts. Each forces a decision — write the position down before moving on; the act of committing is the exercise.

PROMPT 1

Section 4 showed the feeder link's clear-sky capacity is bandwidth-limited because the gateway's huge G/T saturates the MODCOD ladder — the same saturation m5-7 found for high-G/T terminals. This means the gateway's enormous sensitivity is "wasted" in clear sky (it buys no extra clear-sky capacity). Yet nobody builds a smaller, cheaper gateway. Reconcile this: if the G/T is wasted in clear sky, what is the large dish actually buying, and where does its value appear? Connect the answer to m8-2's rain problem and decide whether the gateway is sized for clear-sky capacity or for rain-fade survival — and what that says about how to think about margin on a concentration point.

PROMPT 2

Section 5 argued the feeder link is not just a big user link because of concentration, band-driven fragility, and siting freedom. But there is a design choice hidden in "concentration": a satellite could funnel all its traffic through one feeder link (maximum concentration, maximum failure consequence) or split it across several feeder links to different gateways (less concentration, more resilience, more feeder hardware). Work the trade: what does splitting a satellite's feeder across two gateways buy in availability, and cost in feeder hardware and complexity? Decide whether the concentration is a bug to be mitigated or a feature to be embraced, and how m8-2's site diversity changes the calculation.

PROMPT 3

Section 2 noted feeder links push to Q/V-band for bandwidth despite the rain penalty. This is a bet: that the extra bandwidth is worth the extra fragility, mitigated by site diversity. But the bet's payoff depends on the constellation's traffic and climate. Construct the argument for and against Q/V-band feeders for two different operators: one serving dense traffic in wet tropical markets, one serving moderate traffic in dry temperate markets. For which does the Q/V-band bandwidth justify its rain fragility, and what does the answer say about whether feeder-band choice is a constellation-wide decision or a per-gateway one?

PROMPT 4

This lesson built the feeder link entirely from the clear-sky budget, explicitly deferring availability to m8-2. But the clear-sky capacity (40–53 Gbps) is the number that will size the whole ground segment — how many gateways, how much backhaul, what it costs. If m8-2 shows that rain forces the feeder to run at a lower average rate (backing off MODCOD during fades), the clear-sky number overstates the deliverable capacity. Apply the load-bearing-assumption discipline: is the clear-sky feeder capacity the right number to size the ground segment on, or should it be an availability-weighted average? Specify what m8-2 must compute before the ground-segment sizing (m8-7) can trust the feeder capacity, and predict roughly how much the availability weighting will reduce it.
