3. Being a Good API Citizen — What Xavier Needs From CDEs

Xavier's team maintains the Cockpit API. CDEs are its primary internal users. A CDE who misuses the API creates support burden. A CDE who uses it correctly — and provides useful feedback — helps Xavier's team improve it.

What good API use looks like

  • Always check the API documentation for the intended use of each endpoint before building a client
  • Respect rate limits — build clients with backoff, not blind retry loops
  • Use the right HTTP verb for the right operation — don't use GET for operations that have side effects
  • Handle all documented error codes, including 4xx (client errors you should fix) and 5xx (server errors you should retry)
  • Version your API calls — if the Cockpit API releases v2 of an endpoint, don't silently keep calling v1 until it breaks

What good bug reports look like

Xavier's team cannot fix bugs they cannot reproduce. A useful bug report contains: the exact endpoint and HTTP method called, the request body/params, the response status and body, the timestamp (so Xavier can check server logs), whether it's reproducible or intermittent, and the business impact (what was the customer trying to do, what could they not do).

THE SCENARIO XAVIER MIGHT POSE "A CDE tells you: 'The Cockpit API is returning wrong data for my customer's mission.' What information do you gather before escalating to my team?" Expected answer: endpoint and params, response body, timestamp, whether it's consistent or intermittent, comparison against what the data should be (and how you know), whether it affects all customers or just this one.

4. Software Architecture Concepts to Be Fluent In

Xavier will not quiz you on implementing a B-tree or designing a distributed system. But he will probe whether you understand how the systems around you work. These are the concepts where gaps will show.

Service boundaries and microservices

Loft's ground software is a set of services — Cockpit, MOS, ground station management, data management — each with a defined API boundary. A failure in one service does not necessarily propagate to others (this is the point of the boundaries). When you diagnose a system problem, you need to determine which service is the source: is the issue in the API gateway, in the service that processes the command, or in the downstream service it calls?

Containers and Docker basics

A container packages an application with all its dependencies so it runs identically in any environment. Key concepts: an image is the template; a container is a running instance; a Dockerfile defines how to build the image. Kubernetes orchestrates containers — it decides which host runs which container, restarts failed containers, and scales up replicas when load increases. A CDE deploying a custom monitoring script to Loft's infrastructure needs to be able to write a basic Dockerfile.

MINIMAL DOCKERFILE — CDE MONITORING TOOL FROM python:3.12-slim WORKDIR /app COPY requirements.txt . RUN pip install --no-cache-dir -r requirements.txt COPY mission_monitor.py . # Non-root user for security RUN useradd -m monitoruser USER monitoruser CMD ["python", "mission_monitor.py"]

Message queues — why Kafka matters

Satellite telemetry arrives at high frequency and must be processed by multiple consumers: the Cockpit dashboard, the alert system, the telemetry database writer, CDE custom tools. A message queue (Kafka) decouples producers (ground station receivers) from consumers. Each consumer processes at its own rate; if one consumer is slow, it doesn't slow the others. For a CDE building a tool that processes telemetry, this means: subscribe to the relevant Kafka topic, consume at your own rate, commit offsets when you've processed each message.

WebSockets for real-time telemetry

Cockpit's real-time dashboard uses WebSocket to push telemetry updates to the browser as they arrive. REST (request/response) is insufficient for real-time: polling every 500ms creates unnecessary load. WebSocket maintains a persistent connection and the server pushes updates when data changes. A CDE building a custom real-time dashboard should use WebSocket, not a REST polling loop.

Exercise 1 — Kafka Telemetry Consumer

Consume telemetry from a Kafka topic, process records in a sliding window, and detect anomalies. This is the pattern for a CDE tool that subscribes to Loft's telemetry stream without polling the REST API.

# pip install kafka-python (or confluent-kafka for production)
# This exercise uses a mock consumer for portability

from collections import deque
from dataclasses import dataclass, field
from datetime import datetime
from typing import Iterator
import json, time, random

# --- Mock Kafka infrastructure ---
@dataclass
class KafkaMessage:
    topic: str
    partition: int
    offset: int
    key: bytes
    value: bytes
    timestamp: float

def mock_kafka_consumer(topic: str, n_messages: int = 100) -> Iterator[KafkaMessage]:
    """Simulate consuming from a Kafka topic."""
    channels = {
        'EPS_BATT_SOC':   (80, 95, 0.5),     # (mean, max, std)
        'THERM_PAYLOAD':  (24, 55, 1.0),
        'ADCS_ANGLE_ERR': (0.3, 2.0, 0.1),
        'COMMS_RSSI':     (-72, -50, 3.0),
    }
    # Inject an anomaly window
    anomaly_start, anomaly_end = 60, 80
    for i in range(n_messages):
        ch = list(channels.keys())[i % len(channels)]
        mean, max_val, std = channels[ch]
        if ch == 'EPS_BATT_SOC' and anomaly_start <= i < anomaly_end:
            val = random.gauss(18, 2)  # Battery drain anomaly
        else:
            val = min(max_val, random.gauss(mean, std))
        payload = json.dumps({
            "mission_id": "YAM9-PAYLOAD-A",
            "channel": ch,
            "value": round(val, 3),
            "unit": {"EPS_BATT_SOC":"%","THERM_PAYLOAD":"degC","ADCS_ANGLE_ERR":"deg","COMMS_RSSI":"dBm"}[ch],
            "ts": datetime.utcnow().isoformat() + "Z"
        }).encode()
        yield KafkaMessage(
            topic=topic, partition=0, offset=i,
            key=ch.encode(), value=payload,
            timestamp=time.time()
        )
        time.sleep(0.01)  # 10ms between messages = 100 msgs/sec

# --- Sliding window anomaly detector ---
@dataclass
class ChannelMonitor:
    channel: str
    window_size: int = 20         # number of samples in sliding window
    warn_threshold: float = 2.5   # std deviations for warning
    crit_threshold: float = 4.0   # std deviations for critical
    values: deque = field(default_factory=lambda: deque(maxlen=20))
    alerts: list = field(default_factory=list)

    def ingest(self, value: float, timestamp: str) -> dict | None:
        self.values.append(value)
        if len(self.values) < self.window_size // 2:
            return None  # Not enough data to assess
        mean = sum(self.values) / len(self.values)
        variance = sum((v - mean)**2 for v in self.values) / len(self.values)
        std = variance ** 0.5
        if std < 1e-6:
            return None  # Zero variance — constant signal
        z_score = abs(value - mean) / std
        if z_score >= self.crit_threshold:
            return {"severity":"CRITICAL","channel":self.channel,
                    "value":value,"mean":round(mean,2),"z":round(z_score,1),
                    "ts":timestamp}
        if z_score >= self.warn_threshold:
            return {"severity":"WARNING","channel":self.channel,
                    "value":value,"mean":round(mean,2),"z":round(z_score,1),
                    "ts":timestamp}
        return None

def run_telemetry_consumer(topic: str = "loft.telemetry.yam9"):
    """Main consumer loop."""
    monitors: dict[str, ChannelMonitor] = {}
    all_alerts = []
    processed = 0

    print(f"Consuming from topic: {topic}")
    print(f"{'='*55}")

    for msg in mock_kafka_consumer(topic, n_messages=100):
        payload = json.loads(msg.value.decode())
        ch = payload["channel"]
        val = payload["value"]
        ts  = payload["ts"]

        # Get or create monitor for this channel
        if ch not in monitors:
            monitors[ch] = ChannelMonitor(channel=ch)

        alert = monitors[ch].ingest(val, ts)
        if alert:
            all_alerts.append(alert)
            icon = "🔴" if alert["severity"] == "CRITICAL" else "🟡"
            print(f"{icon} [{alert['severity']:8s}] {ch}: {val:.2f} "
                  f"(mean={alert['mean']:.2f}, z={alert['z']:.1f}σ) @ {ts}")

        processed += 1

    print(f"\n{'='*55}")
    print(f"Processed: {processed} messages")
    print(f"Alerts: {len(all_alerts)} "
          f"({sum(1 for a in all_alerts if a['severity']=='CRITICAL')} CRITICAL, "
          f"{sum(1 for a in all_alerts if a['severity']=='WARNING')} WARNING)")
    return all_alerts

alerts = run_telemetry_consumer()

Exercise 2 — Structured Bug Report Generator

Build a tool that captures an API interaction and automatically formats a structured bug report in the format Xavier's team needs. Shows you understand what makes a useful bug report.

import json, time
from datetime import datetime
from dataclasses import dataclass, field
from typing import Any

@dataclass
class APIInteraction:
    """Captures a complete API interaction for bug reporting."""
    method: str
    endpoint: str
    request_headers: dict
    request_body: Any
    response_status: int
    response_body: Any
    response_headers: dict
    duration_ms: float
    timestamp: str
    mission_id: str | None = None
    customer_id: str | None = None

@dataclass
class BugReport:
    title: str
    interaction: APIInteraction
    expected_behaviour: str
    actual_behaviour: str
    business_impact: str
    reproducible: bool
    reproduction_steps: list[str]
    reporter: str = "CDE"
    priority: str = "P2"  # P1=blocking, P2=high, P3=medium, P4=low

    def format(self) -> str:
        lines = [
            f"BUG REPORT — {self.title}",
            f"Reporter: {self.reporter} | Priority: {self.priority}",
            f"{'='*60}",
            "",
            "SUMMARY",
            f"  {self.title}",
            "",
            "API INTERACTION",
            f"  Timestamp:  {self.interaction.timestamp}",
            f"  Method:     {self.interaction.method}",
            f"  Endpoint:   {self.interaction.endpoint}",
            f"  Mission:    {self.interaction.mission_id or 'N/A'}",
            f"  Customer:   {self.interaction.customer_id or 'N/A'}",
            f"  Duration:   {self.interaction.duration_ms:.0f}ms",
            "",
            "REQUEST",
            f"  Headers (sanitised):",
        ]
        # Sanitise auth headers
        safe_headers = {k: ("***" if "auth" in k.lower() else v)
                        for k, v in self.interaction.request_headers.items()}
        for k, v in safe_headers.items():
            lines.append(f"    {k}: {v}")
        if self.interaction.request_body:
            lines += ["  Body:", f"    {json.dumps(self.interaction.request_body, indent=4)}"]
        lines += [
            "",
            "RESPONSE",
            f"  Status: {self.interaction.response_status}",
            f"  Body:   {json.dumps(self.interaction.response_body, indent=4)}",
            "",
            "EXPECTED VS ACTUAL",
            f"  Expected: {self.expected_behaviour}",
            f"  Actual:   {self.actual_behaviour}",
            "",
            "BUSINESS IMPACT",
            f"  {self.business_impact}",
            "",
            f"REPRODUCIBLE: {'YES' if self.reproducible else 'NO — INTERMITTENT'}",
            "",
            "REPRODUCTION STEPS",
        ]
        for i, step in enumerate(self.reproduction_steps, 1):
            lines.append(f"  {i}. {step}")
        return "\n".join(lines)
