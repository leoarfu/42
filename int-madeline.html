2. Payload Commissioning — What a COE/CDE Does

Commissioning is the period between launch and the start of normal operations during which the satellite, Hub, and customer payloads are brought up and validated. The CDE/COE leads the payload side of this process.

The commissioning sequence

  1. Bus and Hub commissioning — Done by Loft's flight ops team. Verifies satellite attitude, power, thermal, and comms. The CDE monitors but does not lead this phase.
  2. Hub-payload interface checkout — First power-on of the payload through the Hub. Verify the Hub sees the payload, the electrical interface is nominal, and the commanding chain works. This is the first CDE-led step.
  3. Payload functional test — Execute the pre-defined commissioning test sequence from the ConOps. Each test has a pass/fail criterion. The CDE executes the commands, records the responses, and compares against expected values.
  4. First light / mission data collection — The payload performs its first operational task. For an imager: first image capture and downlink. For an SDR: first spectrum observation. The CDE and customer evaluate quality against mission acceptance criteria.
  5. Mission acceptance — Customer formally accepts the mission as commissioned. The CDE documents findings, deviations from the ConOps, and any constraints on operations. Mission transitions to on-orbit steady-state.

What goes wrong during commissioning

The most common commissioning anomalies are: unexpected payload behaviour on first power-on (usually a firmware or configuration issue rather than hardware damage), interface mismatches between the Hub and the payload that were not caught during ground integration, and discrepancies between the ground-tested ConOps and the actual on-orbit environment (e.g., thermal behaviour that differs from thermal vacuum test).

THE KEY COMMISSIONING QUESTION MADELINE MIGHT ASK "During commissioning, your payload test sequence produces a result that doesn't match the expected value in the ConOps — but the payload appears otherwise healthy. What do you do?" Answer: Do not immediately conclude the hardware is faulty. First, check: (1) is the test procedure itself correct — were the commands sent in the right order, at the right timing? (2) Is the expected value in the ConOps definitely right — was it set from ground test data under equivalent conditions? (3) Is the observed value within any acceptable tolerance? Only after these checks do you escalate to a hardware investigation.

Pattern 2: Stateful alert suppression

Fire an alert when a condition first appears. Suppress repeated alerts while the condition persists. Re-arm when the condition clears. This prevents alert storms from a single persistent issue.

4. REST API Interaction — The Cockpit API Model

The Cockpit platform exposes a REST API for advanced customers and for CDEs building tooling on top of it. A CDE who understands API design can write better integration scripts, file better bug reports to Xavier's team, and help customers integrate Cockpit into their own tools.

HTTP verbs and their satellite ops semantics

  • GET /missions/{id}/telemetry — Read current telemetry. Safe (no side effect). Cache-friendly. Use for monitoring loops.
  • POST /missions/{id}/commands — Submit a command to the spacecraft. Not idempotent — two identical POSTs send two commands. Always check for duplicate-command guards in your client code.
  • PUT /missions/{id}/schedule — Replace the mission schedule. Idempotent — sending the same schedule twice has the same result as once. Use for configuration updates.
  • PATCH /missions/{id}/config — Partial update. More efficient than PUT when changing one field of a large configuration object.

Authentication patterns

Cockpit uses bearer token authentication. The token is obtained via a credentials exchange (POST to /auth/token) and has an expiry. Your client code must: handle token expiry gracefully (re-authenticate transparently rather than crashing), never log the token, and store it in memory or a secrets manager rather than in code or on disk.

Pagination and large telemetry datasets

A mission with months of telemetry history cannot return all records in one response. Cockpit's API (like most production APIs) uses cursor-based or offset-based pagination. Your client must handle pagination — if you only read the first page, you'll miss data and build incorrect analyses.

WHAT MADELINE WANTS TO HEAR ABOUT APIS Not "I've used REST APIs." She wants: "I understand idempotency — I'm careful about POST vs PUT especially for commanding. I handle token expiry transparently. I paginate. I handle 429 rate limit responses with backoff. I never trust that a 200 response means what I expect without checking the response body." That level of specificity is what separates operational Python from casual scripting.

5. Customer Communication During Commissioning Anomalies

Commissioning anomalies are harder to communicate than on-orbit anomalies because there is more uncertainty. You don't yet have a baseline for "normal" on this specific mission. The customer is anxious — they've been waiting months for launch and commissioning. The last thing they want to hear is "we're not sure what's happening."

The principle: factual honesty over false reassurance

Madeline's mathematical background makes her particularly sensitive to imprecise communication. If the telemetry shows a payload temperature 8°C above the expected value but within limits, say exactly that — don't say "temperatures look a bit high but we're watching it." Give the number. State the limit. State the margin. State whether you're concerned or not and why.

Separating known from unknown

In any customer commissioning update, be explicit about what is known vs unknown. "We know: the Hub-payload interface is healthy and all commanding is responding nominally. We don't yet know: whether the first-light data quality meets your acceptance threshold — we need the next pass downlink to assess. We expect to have that data by [specific time]."

MADELINE'S LIKELY SCENARIO QUESTION "You're commissioning a customer's imaging payload. First light produces images, but the resolution is noticeably worse than the ground test. The customer is on a call with you. What do you say and do?" Answer structure: (1) Describe what you observe precisely — quantify the degradation. (2) List the possible causes without jumping to conclusions: thermal settling, focus shift, contamination, calibration difference. (3) State what data you need to distinguish between causes and when you'll have it. (4) Do not promise a fix you cannot deliver. Do commit to a timeline for your next diagnostic step.

Exercise 1 — Cockpit API Client with Retry and Pagination

Build a production-grade API client class that handles authentication, token refresh, rate limiting, and paginated telemetry fetching. This is what a CDE actually submits to Madeline's team as a tooling contribution.

import time, logging, json
from typing import Generator, Any
from dataclasses import dataclass
from unittest.mock import MagicMock, patch
import requests

logger = logging.getLogger("cockpit_client")

@dataclass
class CockpitConfig:
    base_url: str
    client_id: str
    client_secret: str
    timeout_s: int = 10
    max_retries: int = 3
    page_size: int = 100

class CockpitAPIError(Exception):
    def __init__(self, status_code: int, message: str):
        self.status_code = status_code
        super().__init__(f"HTTP {status_code}: {message}")

class CockpitClient:
    """Production-grade client for Loft Orbital Cockpit API."""

    def __init__(self, config: CockpitConfig):
        self.config = config
        self._token: str | None = None
        self._token_expires_at: float = 0
        self._session = requests.Session()

    # --- Auth ---
    def _authenticate(self) -> None:
        """Obtain or refresh bearer token."""
        resp = self._session.post(
            f"{self.config.base_url}/auth/token",
            json={"client_id": self.config.client_id,
                  "client_secret": self.config.client_secret},
            timeout=self.config.timeout_s
        )
        resp.raise_for_status()
        data = resp.json()
        self._token = data["access_token"]
        self._token_expires_at = time.time() + data.get("expires_in", 3600) - 60
        logger.info("Authenticated successfully, token valid for ~%ds",
                    data.get("expires_in", 3600))

    def _get_token(self) -> str:
        """Return valid token, re-authenticating if expired."""
        if not self._token or time.time() >= self._token_expires_at:
            self._authenticate()
        return self._token

    # --- Request with retry ---
    def _request(self, method: str, path: str, **kwargs) -> dict:
        url = f"{self.config.base_url}{path}"
        headers = {"Authorization": f"Bearer {self._get_token()}",
                   "Content-Type": "application/json"}
        delay = 1.0
        last_error = None

        for attempt in range(self.config.max_retries):
            try:
                resp = self._session.request(
                    method, url, headers=headers,
                    timeout=self.config.timeout_s, **kwargs
                )
                # Handle rate limiting
                if resp.status_code == 429:
                    wait = int(resp.headers.get("Retry-After", delay * 2))
                    logger.warning("Rate limited. Waiting %ds (attempt %d)", wait, attempt + 1)
                    time.sleep(wait)
                    delay *= 2
                    continue
                # Handle auth expiry mid-session
                if resp.status_code == 401:
                    logger.info("Token rejected, re-authenticating...")
                    self._token = None
                    headers["Authorization"] = f"Bearer {self._get_token()}"
                    continue
                # Permanent errors — don't retry
                if 400 <= resp.status_code < 500 and resp.status_code not in (429, 401):
                    raise CockpitAPIError(resp.status_code, resp.text[:200])
                # Server errors — retry
                if resp.status_code >= 500:
                    raise requests.HTTPError(f"Server error {resp.status_code}")

                resp.raise_for_status()
                return resp.json()

            except (requests.ConnectionError, requests.Timeout, requests.HTTPError) as e:
                last_error = e
                if attempt < self.config.max_retries - 1:
                    logger.warning("Request failed (attempt %d/%d): %s. Retry in %.1fs",
                                   attempt + 1, self.config.max_retries, e, delay)
                    time.sleep(delay)
                    delay *= 2

        raise last_error or RuntimeError("Request failed after all retries")

    # --- Paginated telemetry fetch ---
    def get_telemetry(self, mission_id: str,
                      start_iso: str, end_iso: str,
                      channels: list[str] | None = None) -> Generator[dict, None, None]:
        """
        Yield all telemetry records for a mission in a time window.
        Handles pagination transparently.
        """
        params = {"start": start_iso, "end": end_iso,
                  "page_size": self.config.page_size}
        if channels:
            params["channels"] = ",".join(channels)

        cursor = None
        page_num = 0
        total_yielded = 0

        while True:
            if cursor:
                params["cursor"] = cursor
            data = self._request("GET", f"/missions/{mission_id}/telemetry",
                                 params=params)
            records = data.get("records", [])
            for record in records:
                yield record
                total_yielded += 1
            page_num += 1
            cursor = data.get("next_cursor")
            logger.debug("Page %d: %d records (total so far: %d)",
                         page_num, len(records), total_yielded)
            if not cursor:
                logger.info("Telemetry fetch complete: %d records in %d pages",
                            total_yielded, page_num)
                break

    # --- Command with verification ---
    def send_command(self, mission_id: str, command: dict,
                     verify_channel: str | None = None,
                     verify_timeout_s: int = 120) -> dict:
        """
        Send a command and optionally wait for telemetry confirmation.
        Returns the command submission receipt.
        """
        logger.info("Sending command to %s: %s", mission_id, command.get("command_id"))
        receipt = self._request("POST", f"/missions/{mission_id}/commands",
                                json=command)
        cmd_id = receipt.get("submission_id")
        logger.info("Command submitted: %s", cmd_id)

        if verify_channel:
            logger.info("Waiting up to %ds for telemetry confirmation on %s...",
                        verify_timeout_s, verify_channel)
            # In practice: poll telemetry until the expected value appears or timeout
            # This is left as an exercise — see py-challenge below
            pass

        return receipt

Exercise 2 — Commissioning Test Runner

Execute a sequence of commissioning tests defined in a YAML-like structure, record pass/fail against expected values, and generate an acceptance report. Models the kind of tooling a CDE builds to automate commissioning procedures.

from dataclasses import dataclass, field
from typing import Callable, Any
from datetime import datetime
import json, random

@dataclass
class CommissioningTest:
    test_id: str
    description: str
    command: dict               # What to send
    expected_channel: str       # Telemetry channel to check
    expected_value: Any         # Expected telemetry value
    tolerance: float = 0.0      # Acceptable deviation (for floats)
    critical: bool = False      # If True, failure halts sequence

@dataclass
class TestResult:
    test_id: str
    description: str
    status: str           # PASS | FAIL | SKIP
    actual_value: Any
    expected_value: Any
    timestamp: str
    notes: str = ""

def run_commissioning_sequence(
    tests: list[CommissioningTest],
    send_command_fn: Callable,
    read_telemetry_fn: Callable,
    mission_id: str
) -> list[TestResult]:
    """Execute commissioning test sequence, returning results."""
    results = []
    halted = False

    for test in tests:
        if halted:
            results.append(TestResult(
                test_id=test.test_id, description=test.description,
                status="SKIP", actual_value=None,
                expected_value=test.expected_value,
                timestamp=datetime.utcnow().isoformat()+"Z",
                notes="Skipped — previous critical test failed"
            ))
            continue

        print(f"  Running {test.test_id}: {test.description}...")
        send_command_fn(mission_id, test.command)
        actual = read_telemetry_fn(mission_id, test.expected_channel)

        # Evaluate pass/fail
        if isinstance(test.expected_value, float) and test.tolerance > 0:
            passed = abs(actual - test.expected_value) <= test.tolerance
        else:
            passed = actual == test.expected_value

        status = "PASS" if passed else "FAIL"
        notes = "" if passed else f"Expected {test.expected_value}, got {actual}"

        result = TestResult(
            test_id=test.test_id, description=test.description,
            status=status, actual_value=actual,
            expected_value=test.expected_value,
            timestamp=datetime.utcnow().isoformat()+"Z",
            notes=notes
        )
        results.append(result)
        print(f"    → {status}" + (f": {notes}" if notes else ""))

        if not passed and test.critical:
            print(f"  ⚠ CRITICAL test failed — halting sequence.")
            halted = True

    return results

def generate_acceptance_report(mission_id: str, results: list[TestResult]) -> str:
    passed = sum(1 for r in results if r.status == "PASS")
    failed = sum(1 for r in results if r.status == "FAIL")
    skipped = sum(1 for r in results if r.status == "SKIP")
    lines = [
        f"COMMISSIONING ACCEPTANCE REPORT",
        f"Mission: {mission_id}",
        f"Generated: {datetime.utcnow().isoformat()}Z",
        f"{'='*55}",
        f"Results: {passed} PASS  |  {failed} FAIL  |  {skipped} SKIP",
        f"Verdict: {'ACCEPTED' if failed == 0 else 'NOT ACCEPTED — SEE FAILURES'}",
        "",
    ]
    for r in results:
        icon = {'PASS':'✓','FAIL':'✗','SKIP':'○'}[r.status]
        lines.append(f"  {icon} [{r.status:4s}] {r.test_id}: {r.description}")
        if r.notes:
            lines.append(f"         ↳ {r.notes}")
    return "\n".join(lines)
